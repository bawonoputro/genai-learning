# Fine-Tuning and Quantization of GPT-2 (Steam Game Descriptions)

This project fine-tunes **GPT-2** on Steam game description text, evaluates baseline vs fine-tuned generation, and applies quantization with two approaches:

1. **PyTorch dynamic quantization (eager mode)**  
2. **ONNX Runtime dynamic INT8 quantization** (for true size on disk comparison)

---

## Notebook Workflow

### 1) Baseline (`notebooks/1_baseline.ipynb`)
- Loads `gpt2`
- Generates text on 5 prompts
- Measures average generation latency
- Saves:
  - `results/baseline_outputs.txt`
  - baseline row in `results/metrics.csv`

### 2) Fine-tuning (`notebooks/2_finetuning.ipynb`)
- Loads train/validation text datasets
- Tokenizes and fine-tunes GPT-2 with `Trainer`
- Saves fine-tuned checkpoint to:
  - `models/finetuned-gpt2/`
- Generates outputs and saves:
  - `results/finetuned_outputs.txt`
  - fine-tuned row in `results/metrics.csv`

### 3) Quantization (`notebooks/3_quantization.ipynb`)
- Applies PyTorch eager dynamic quantization
- Exports fine-tuned model to ONNX (FP32)
- Quantizes ONNX model to INT8
- Saves ONNX quantization results to:
  - `results/onnx_quantization_summary.csv`
  - ONNX rows in `results/metrics.csv`

## Key Results

From `3_quantization.ipynb`:

- **FP32 ONNX size:** 475.13 MB  
- **INT8 ONNX size:** 119.69 MB  
- **Compression ratio (FP32/INT8):** 3.97x  
- **Size reduction:** 74.81%

---
### Why did PyTorch eager quantization produce a larger saved file?

PyTorch eager dynamic quantization optimizes model computation (mainly for CPU runtime), but the serialized checkpoint size is not guaranteed to shrink.

In this project, the quantized eager artifact became larger than the FP32 safetensors file possibly due to a combination of:

- **Extra quantization metadata** (e.g. scales and related state).
- **Serialization/container overhead differences** (`.bin` pickle-based format vs `.safetensors` layout).
- **Runtime optimization is not equivalent to storage optimization**: eager quantization targets execution behavior such as latency, not necessarily minimal disk footprint.

Because of this, ONNX FP32 vs ONNX INT8 was used for a more reliable size on disk comparison.

## Why ONNX for size comparison?

For a clearer size comparison, this project uses:
- `gpt2_fp32.onnx` vs `gpt2_int8.onnx`
